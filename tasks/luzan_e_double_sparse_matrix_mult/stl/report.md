## 1. Контекст

В данной работе реализация умножения разреженных матриц переносится на модель параллелизма `std::thread`.

Стандартная библиотека C++ не предоставляет встроенного scheduler-а или автоматического распределения задач.
Поэтому вся организация параллельного выполнения реализуется вручную:

- разбиение диапазонов данных;
- создание потоков;
- распределение работы;
- ожидание завершения вычислений;
- объединение результатов.

Такая реализация позволяет напрямую контролировать структуру вычислений и накладные расходы thread-based модели.

Последовательная версия из `seq/report.md` используется как baseline для проверки корректности и вычисления ускорения.

## 2. Постановка задачи

Требуется реализовать параллельное умножение двух разреженных матриц:
$$  
A \in \mathbb{R}^{M \times K}, \quad  
B \in \mathbb{R}^{K \times N}  
$$
представленных в формате CCS.

Результирующая матрица:
$$  
C = A \times B  
$$
также должна быть сохранена в формате CCS.

Подробное описание структуры данных, ограничений и последовательного алгоритма приведено в `seq/report.md`.

Основная цель STL-версии - реализовать ручное распараллеливание без использования runtime-систем высокого уровня.

---

## 3. Базовый алгоритм

Последовательная версия вычисляет результирующую матрицу по столбцам.
Для каждого столбца матрицы (B):

1. создаётся временный аккумулятор;
2. перебираются ненулевые элементы столбца;
3. выполняется накопление произведений;
4. ненулевые элементы сохраняются в CCS-структуру результата.

Поскольку вычисления отдельных столбцов независимы, внешний цикл по столбцам используется как точка распараллеливания.

---

## 4. Схема распараллеливания

### Разбиение диапазонов

Диапазон столбцов матрицы (B) делится между потоками статически.
Размер блока вычисляется как:

chunk = $\left\lceil \frac{cols_B}{num\_threads} \right\rceil$

Для потока с номером `thread_id` диапазон определяется как:

```cpp
unsigned start = thread_id * chunk;
unsigned end =
    std::min(start + chunk,
             static_cast<unsigned>(b.cols));
```

Каждый поток обрабатывает собственный непрерывный диапазон столбцов.

### Работа потока

Каждый поток:

1. получает диапазон столбцов;
2. последовательно вычисляет принадлежащие ему столбцы;
3. сохраняет результаты в собственные ячейки промежуточных структур.

Основной worker:

```cpp
auto worker = [&](unsigned thread_id) {
  unsigned start = thread_id * chunk;
  unsigned end = std::min(start + chunk,
                          static_cast<unsigned>(b.cols));

  for (unsigned b_col = start;
       b_col < end;
       ++b_col) {
    ProcessColumn(...);
  }
};
```

### Локальные результаты

Для каждого столбца используются отдельные промежуточные контейнеры:

```cpp
std::vector<std::vector<double>> values_per_col;
std::vector<std::vector<unsigned>> rows_per_col;
```

Внутри `ProcessColumn` создаётся локальный аккумулятор:

```cpp
std::vector<double> tmp_col(a.rows, 0.0);
```

Он полностью принадлежит текущему потоку и не разделяется между потоками выполнения.

### Create - Work - Join

Потоки сначала создаются в отдельном цикле:

```cpp
for (unsigned thrd = 0;
     thrd < num_threads;
     ++thrd) {
  threads.emplace_back(worker, thrd);
}
```

После запуска всех потоков выполняется отдельный этап ожидания:

```cpp
for (auto& thrd : threads) {
  thrd.join();
}
```

Такой порядок обеспечивает реальное параллельное выполнение.

### Синхронизация

Дополнительные механизмы синхронизации не используются.

Причина - отсутствие совместной записи в общие данные.

Каждый поток:

- использует собственный `tmp_col`;
- записывает результаты только в свои диапазоны столбцов.

### Потокобезопасность

Гонки данных предотвращаются за счёт:

- полного разделения диапазонов столбцов;
- локальных временных аккумуляторов;
- отсутствия конкурентной записи в одну и ту же структуру.

## 5. Детали реализации

### Структура файлов

- `stl/include/ops_stl.hpp` - объявление backend;
- `stl/src/ops_stl.cpp` - реализация ручного распараллеливания.

### Диапазоны

Поток `t` обрабатывает диапазон:
$$t \cdot chunk,\ \min((t+1)\cdot chunk,\ cols_B))$$

Таким образом:

- диапазоны не пересекаются;
- каждый столбец вычисляется ровно один раз;
- нагрузка распределяется статически.

### Создание потоков

Используется:

```cpp
std::thread
```

Количество потоков:

```cpp
const unsigned num_threads =
    std::thread::hardware_concurrency();
```

### Объединение результатов

После завершения всех потоков выполняется последовательная сборка:

```cpp
AssembleResult(...)
```

Финальная CCS-структура формируется в один поток.

### Работа с памятью

Дополнительная память расходуется на:

- локальные аккумуляторы потоков;
- промежуточные структуры столбцов;
- объекты `std::thread`.

Каждый поток многократно создаёт `tmp_col`.
Это увеличивает число динамических аллокаций по сравнению с TBB-версией, использующей thread-local storage.

### Анализ результатов

На двух потоках STL-версия показывает почти линейное ускорение.

При увеличении числа потоков до четырёх эффективность снижается.

Основные причины:

- накладные расходы на создание потоков;
- стоимость `join`;
- повторные аллокации локальных аккумуляторов;
- отсутствие динамической балансировки нагрузки.
  
### Накладные расходы create/join

Каждый запуск вычислений требует:

1. создания объектов потоков;
2. запуска потоков ОС;
3. ожидания завершения через `join`.

При небольших задачах эти расходы могут становиться сопоставимыми со временем самих вычислений.

Ручная реализация на `std::thread` позволила напрямую контролировать структуру распараллеливания и организацию памяти.

Основные преимущества подхода:

- полный контроль над распределением диапазонов;
- отсутствие скрытого runtime;
- прозрачная схема вычислений.

Основные ограничения:

- необходимость ручного управления потоками;
- отсутствие встроенной балансировки нагрузки;
- более высокие накладные расходы create/join;
- большее количество ручной работы по обеспечению потокобезопасности.

Для задач с регулярной структурой нагрузки статическое разбиение диапазонов работает эффективно.
Однако при неравномерной разреженности балансировка может ухудшаться.
