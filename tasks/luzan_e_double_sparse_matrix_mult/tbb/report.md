## 1. Контекст

В данной работе последовательная реализация умножения разреженных матриц в формате CCS
переносится на модель параллелизма oneTBB.

Задача хорошо подходит для task-based распараллеливания,
поскольку вычисление различных столбцов результирующей матрицы может выполняться независимо.
Вместо ручного управления потоками oneTBB предоставляет runtime-планировщик задач,
автоматически распределяющий работу между worker-потоками.

По сравнению с OpenMP библиотека oneTBB предоставляет:

- более гибкую модель декомпозиции задач;
- динамическую балансировку нагрузки;
- встроенные механизмы управления конкуренцией;
- специализированные инструменты для локального хранения данных потоков.

Последовательная версия из `seq/report.md` используется как baseline для проверки корректности и вычисления ускорения.

## 2. Постановка задачи

Требуется реализовать параллельное умножение двух разреженных матриц:

$$  
A \in \mathbb{R}^{M \times K}, \quad  
B \in \mathbb{R}^{K \times N}  
$$
представленных в формате CCS.

Результирующая матрица:
$$  
C = A \times B  
$$
также должна быть сформирована в формате CCS.

Подробное описание структуры данных, ограничений и последовательного алгоритма приведено в `seq/report.md`.

Основная цель TBB-версии — исследовать эффективность task-based распараллеливания при вычислении независимых столбцов результата.

---

## 3. Базовый алгоритм

Последовательная версия вычисляет результирующую матрицу по столбцам.
Для каждого столбца матрицы (B):

1. создаётся временный аккумулятор;
2. перебираются ненулевые элементы столбца;
3. соответствующие элементы матрицы (A) умножаются и накапливаются;
4. ненулевые значения сохраняются в CCS-структуру результата.

Поскольку вычисления отдельных столбцов не зависят друг от друга,
внешний цикл по столбцам является естественной точкой декомпозиции задач.

## 4. Схема распараллеливания

### Используемый примитив

Для распараллеливания используется:

```cpp
oneapi::tbb::parallel_for
```

Данный примитив автоматически разбивает диапазон задач на поддиапазоны и распределяет их между worker-потоками runtime oneTBB.

### Диапазон работы

Диапазон определяется через:

```cpp
oneapi::tbb::blocked_range<unsigned>(0, b.cols)
```

Каждая задача обрабатывает некоторый непрерывный диапазон столбцов результирующей матрицы.

Итерация:

```cpp
for (unsigned b_col = range.begin();
     b_col < range.end();
     ++b_col)
```

вычисляет независимые столбцы результата.

### Grainsize

В реализации используется grainsize по умолчанию.
Поскольку число столбцов матрицы достаточно велико ((2500)),
runtime oneTBB способен автоматически формировать подзадачи подходящего размера без явного задания grainsize.

Слишком маленький grainsize увеличил бы:

- число создаваемых задач;
- накладные расходы scheduler-а;
- стоимость work stealing.

Слишком крупный grainsize мог бы ухудшить балансировку нагрузки между worker-потоками.
Для рассматриваемой задачи автоматическое разбиение оказалось достаточным.

### Partitioner

Используется partitioner по умолчанию (`auto_partitioner`).
Он позволяет runtime oneTBB динамически адаптировать разбиение диапазонов во время выполнения.
А также использовать механизм work stealing для балансировки нагрузки.
Явное использование `static_partitioner` не потребовалось.

### Балансировка нагрузки

Балансировка выполняется автоматически runtime oneTBB.
Scheduler может выдать оставшиеся задачи через механизм work stealing освободившемуся woker-у.

Это особенно важно для задач с потенциально неравномерным распределением ненулевых элементов.

## 5. Детали реализации

### Структура файлов

- `tbb/include/ops_tbb.hpp` — объявление backend;
- `tbb/src/ops_tbb.cpp` — реализация параллельного умножения.

### Основной параллельный участок

```cpp
oneapi::tbb::parallel_for(
    oneapi::tbb::blocked_range<unsigned>(0, b.cols),
    [&](const oneapi::tbb::blocked_range<unsigned>& range) {
      for (unsigned b_col = range.begin();
           b_col < range.end();
           ++b_col) {
        ...
      }
    });
```

### Локальные аккумуляторы

Для хранения временных данных используется:

```cpp
oneapi::tbb::enumerable_thread_specific
```

Каждый worker получает собственный аккумулятор:

```cpp
oneapi::tbb::enumerable_thread_specific<
    std::vector<double>
> tls_tmp;
```

Это позволяет:

- избежать гонок данных;
- избежать повторных аллокаций памяти;
- переиспользовать локальные буферы между задачами.

### Объединение результатов

Каждый столбец сохраняется во временные структуры:

```cpp
values_per_col[b_col]
rows_per_col[b_col]
```

После завершения `parallel_for` выполняется последовательная сборка итоговой CCS-структуры.

### Работа с памятью

Основные дополнительные затраты памяти связаны с:

- локальными аккумуляторами потоков;
- промежуточными векторами столбцов.

Использование `enumerable_thread_specific` уменьшает количество динамических аллокаций.
В отлчиие от создания временного вектора внутри каждой итерации.

### Устранение гонок данных

Параллельные задачи не выполняют совместную запись в общие структуры результата.
Каждый worker:

- использует собственный временный аккумулятор;
- записывает данные только в свой столбец.

Это позволяет избежать использования:

- `mutex`;
- `atomic`;
- `critical section`.

### Анализ результатов

TBB показал высокую эффективность масштабирования на 2 и 4 worker-потоках.

Основные причины хорошей производительности:

- независимость вычислений по столбцам;
- автоматическая балансировка нагрузки;
- низкие накладные расходы scheduler-а;
- переиспользование локальных аккумуляторов.

### Overhead

Основные источники overhead:

- создание и управление задачами runtime;
- последовательная сборка итоговой CCS-структуры;
- работа с памятью при сохранении промежуточных столбцов.

При слишком мелком grainsize overhead scheduler-а мог бы стать заметнее.

Библиотека oneTBB позволила эффективно распараллелить вычисление произведения разреженных матриц.
Task-based модель успешно справилась с задачей.
Наиболее полезными возможностями oneTBB для данной задачи оказались:

- `parallel_for`;
- `blocked_range`;
- автоматическая балансировка нагрузки;
- `enumerable_thread_specific`.

По сравнению с ручным управлением потоками TBB предоставляет более высокий уровень абстракции.
Встроенный runtime-планировщик задач помогает правильно распределять нагрузку.

Наиболее заметный выигрыш достигается на задачах с достаточно большим числом независимых итераций.
При слишком мелких задачах накладные расходы scheduler-а и разбиения диапазонов могут ограничивать масштабируемость.
