## 1. Контекст

В данной работе последовательная реализация умножения разреженных матриц в формате CCS переносится на модель OpenMP.

Идея распараллеливания - вычисление отдельных столбцов результирующей матрицы не зависит друг от друга.
Это позволит распределить обработку столбцов между несколькими потоками без изменения математической логики алгоритма.

OpenMP выбран как средство распараллеливания благодаря:

- минимальным изменениям последовательного кода;
- поддержке директивного параллелизма;
- встроенным механизмам управления потоками;
- низким накладным расходам для задач shared-memory.

Последовательная версия из `seq/report.md` используется как baseline для проверки корректности и вычисления ускорения.

## 2. Постановка задачи

Требуется реализовать параллельное умножение двух разреженных матриц, представленных в формате CCS.

Входные данные:

$$  
A \in \mathbb{R}^{M \times K}, \quad  
B \in \mathbb{R}^{K \times N}  
$$

Результат:

$$  
C = A \times B  
$$

в формате CCS.

Подробное описание структуры данных, ограничений и базового алгоритма приведено в `seq/report.md`.

Основной целью данной версии является ускорение вычислений за счёт распараллеливания обработки столбцов результирующей матрицы.

---

## 3. Базовый алгоритм

Последовательная версия вычисляет результат столбец за столбцом.
Для каждого столбца матрицы (B):

1. создаётся временный плотный аккумулятор;
2. перебираются ненулевые элементы столбца;
3. соответствующие элементы матрицы (A) умножаются и накапливаются;
4. ненулевые значения сохраняются в результирующую CCS-структуру.

Каждый столбец результата вычисляется независимо, что делает внешний цикл по `b_col` естественной точкой распараллеливания.

---

## 4. Схема распараллеливания

### Параллельная область

Распараллеливается внешний цикл обработки столбцов матрицы (B):

```cpp
#pragma omp parallel for schedule(static)
for (unsigned b_col = 0; b_col < b.cols; b_col++) {
    ...
}
```

Каждая итерация вычисляет один столбец результирующей матрицы.

### Shared-переменные

Общими для всех потоков являются:

- `a`;
- `b`;
- `values_per_col`;
- `rows_per_col`;
- `kEPS`.

Эти структуры либо только читаются, либо каждый поток записывает данные исключительно в собственный индекс столбца.

### Private-переменные

Локальными для каждого потока являются:

- `tmp_col`;
- `b_rows_start`;
- `b_rows_end`;
- `b_pos`;
- `a_pos`;
- промежуточные значения элементов.

Временный аккумулятор `tmp_col` создаётся внутри итерации цикла, поэтому не разделяется между потоками.

### Синхронизация

Гонки данных устраняются за счёт полной изоляции вычислений по столбцам.
Каждый поток записывает результаты только в:

```cpp
values_per_col[b_col]
rows_per_col[b_col]
```

где `b_col` уникален для конкретной итерации.

### Schedule

Используется:

```cpp
schedule(static)
```

Причины выбора:

- тестовые матрицы имеют относительно равномерное распределение ненулевых элементов;
- все итерации имеют близкую вычислительную стоимость;
- статическое распределение создаёт минимальные накладные расходы runtime OpenMP.

### Барьеры

После завершения `parallel for` OpenMP создаёт неявный барьер синхронизации.
После него выполняется последовательная сборка итоговой CCS-структуры.

## 5. Детали реализации

### Структура файлов

- `omp/include/ops_omp.hpp` - объявление OpenMP backend;
- `omp/src/ops_omp.cpp` - реализация параллельного умножения.

### Основные изменения относительно SEQ

По сравнению с последовательной версией были добавлены:

- директива `#pragma omp parallel for`;
- промежуточные буферы:
  - `values_per_col`;
  - `rows_per_col`;
- отдельные локальные аккумуляторы для каждого потока.

### Промежуточные буферы

Непосредственная запись нескольких потоков в:

```cpp
c.value
c.row
```

привела бы к гонкам данных.

Для устранения этой проблемы каждый поток сначала сохраняет результат во временные структуры:

```cpp
std::vector<std::vector<double>> values_per_col;
std::vector<std::vector<unsigned>> rows_per_col;
```

После завершения параллельной области выполняется последовательная сборка итоговой матрицы.

### Устранённые риски гонок

Были устранены следующие потенциальные конфликты:

- одновременная запись в `c.value`;
- одновременная запись в `c.row`;
- совместное использование временного аккумулятора.

Все вычисления внутри итерации цикла полностью независимы.

Распараллеливание внешнего цикла обработки столбцов позволило существенно ускорить вычисление произведения разреженных матриц.
