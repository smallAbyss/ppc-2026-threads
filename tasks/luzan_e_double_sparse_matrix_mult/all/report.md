## 1. Контекст

В данной работе реализована гибридная схема распараллеливания, сочетающая:

- межпроцессный уровень - MPI;
- внутрипроцессный уровень - OpenMP.

Подобная модель используется в высокопроизводительных вычислениях на многопроцессорных системах.
MPI отвечает за распределение данных между процессами, а OpenMP - за загрузку ядер внутри каждого процесса.

Гибридная схема позволяет:

- уменьшить объём работы на один MPI-процесс;
- эффективнее использовать shared-memory внутри узла;
- снизить количество MPI-процессов по сравнению с pure MPI-подходом.

Последовательная версия из `seq/report.md` используется как baseline для проверки корректности и вычисления ускорения.

## 2. Постановка задачи

Требуется реализовать гибридное умножение двух разреженных матриц:

$$  
A \in \mathbb{R}^{M \times K}, \quad  
B \in \mathbb{R}^{K \times N}  
$$

представленных в формате CCS.

Результирующая матрица:
$$  
C = A \times B  
$$
должна быть собрана на процессе с `rank = 0`.

Подробная постановка задачи и описание структуры CCS приведены в корневом отчёте и `seq/report.md`.

Основная цель ALL-версии - исследовать эффективность двухуровневого распараллеливания:

```text
MPI ranks × OpenMP threads
```

---

## 3. Базовый алгоритм

Последовательный алгоритм вычисляет результирующую матрицу по столбцам.

Для каждого столбца матрицы (B):

1. создаётся временный аккумулятор;
2. выполняется накопление произведений;
3. ненулевые значения сохраняются в CCS-структуру результата.

Поскольку вычисление столбцов независимо, задача естественным образом допускает двухуровневое разбиение:

- между MPI-процессами;
- между потоками внутри процесса.

## 4. Межпроцессная схема

### Роли rank-ов

Каждый MPI-процесс получает собственный `rank`:

```cpp
int rank = -1;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
```

и общее число процессов:

```cpp
int nprocs = 0;
MPI_Comm_size(MPI_COMM_WORLD, &nprocs);
```

Процесс с `rank = 0` выполняет:

- начальную валидацию;
- распределение данных;
- сбор итогового результата.

Остальные процессы выполняют только вычислительную часть.

### Распределение данных

Матрицы `A` и `B` рассылаются всем процессам через:

```cpp
BroadcastMatrix(...)
```

внутри которого используется:

```cpp
MPI_Bcast
```

После этого каждый процесс получает полные копии входных матриц.

### Разбиение столбцов

Столбцы матрицы (B) делятся между процессами.

Распределение формируется функцией:

```cpp
BuildColDistribution(...)
```

Каждому rank-у назначаются:

- `local_col_start`;
- `local_col_count`.

Процесс вычисляет только свой диапазон столбцов.

### MPI-вызовы

#### MPI_Bcast

Используется для рассылки входных матриц всем процессам.
Это необходимо, поскольку вычисление каждого столбца требует доступа ко всем данным матриц `A` и `B`.

#### MPI_Gatherv

Используется для сбора информации о числе ненулевых элементов:
Разные rank-и могут вычислять различное число ненулевых элементов.
Поэтому используется именно `Gatherv`, поддерживающий переменные размеры данных.

### Сбор результатов

Локальные столбцы сначала преобразуются в плоские массивы:

```cpp
FlattenLocalCols(...)
```

после чего пересылаются на `rank = 0`.
На корневом процессе формируется итоговая CCS-матрица.

## 5. Внутрипроцессная схема

### Используемая thread-technology

Внутри каждого MPI-процесса используется OpenMP.
Локальный цикл по столбцам распараллеливается через:

```cpp
#pragma omp parallel for
```

### Разделение работы между потоками

Каждый поток OpenMP обрабатывает отдельные столбцы локального диапазона процесса.
Схема:

```text
MPI rank
    OpenMP threads
      local columns
```

Потоки работают независимо друг от друга.

### Локальные аккумуляторы

Каждый поток создаёт собственный временный буфер:

```cpp
std::vector<double> tmp_col(a.rows, 0.0);
```

Буфер является приватным для потока и используется для накопления значений одного столбца.

### Объединение локальных результатов

После завершения OpenMP-цикла локальные столбцы процесса:

- сохраняются в `values_per_col`;
- преобразуются в плоские массивы;
- передаются через MPI на `rank = 0`.

Финальная сборка выполняется только корневым процессом.

### Потенциальные узкие места

Основные ограничения производительности:

#### Коммуникации MPI

Рассылка матриц и сбор результатов создают сетевые накладные расходы.
Особенно заметны:

- `MPI_Bcast`;
- `MPI_Gatherv`.

#### Дублирование памяти

Каждый MPI-процесс хранит собственные копии матриц:
$$  
O(nnz_A + nnz_B)  
$$
что увеличивает потребление памяти.

#### Последовательная сборка

Формирование итоговой CCS-структуры выполняется только на `rank = 0`.
При больших размерах матриц это может становиться bottleneck.

### Интерпретация результатов

- MPI уменьшает объём работы на процесс;

Основные накладные расходы связаны с:

- `MPI_Bcast`;

Эффективность рассчитывается относительно:
total_workers = $ranks \times threads_per_rank$

При увеличении числа workers эффективность снижается из-за:

- стоимости MPI-коммуникаций;
- синхронизации;
- последовательной сборки результата.

## 10. Выводы

Гибридная схема MPI + OpenMP позволяет эффективно комбинировать:

- межпроцессное распределение данных;
- внутрипроцессное многопоточное выполнение.

Подход особенно полезен для крупных задач и вычислительных систем с несколькими узлами.

Основные преимущества:

- иерархическая модель параллелизма;
- хорошая масштабируемость на больших задачах;
- эффективное использование shared-memory внутри rank-а.

Основные ограничения:

- стоимость MPI-коммуникаций;
- дополнительные точки синхронизации;
- дублирование памяти между процессами;
- последовательная сборка результата на `rank = 0`.

Для сравнительно небольших матриц читсые OpenMP или TBB могут показывать лучшую производительность.
Это возможно из-за меньших коммуникационных overhead.
